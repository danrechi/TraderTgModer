## Шаг 1. Разработка инференс-модуля

### Реализованный эндпоинт

Создан файл `app/routers/classify.py`, который реализует микросервис классификации новостей.

**Маршрут:** `POST /ml/classify`

**Тело запроса (JSON):**
```json
{
  "text": "Биткоин обновил исторический максимум выше 100 000 долларов"
}
```

**Ответ:**
```json
{
  "category": "Криптовалюты",
  "confidence": 0.9123,
  "model": "SVM (LinearSVC)",
  "timestamp": "2025-04-10T09:00:00Z"
}
```

**Детали реализации:**
- Классификатор загружается как синглтон при первом запросе (`lazy loading`) — не замедляет старт сервера
- Каждый запрос логируется в `logs/inference.log` в формате:
  ```
  2025-04-10 09:05:13 | INFO | text="Биткоин..." | category=Криптовалюты | confidence=0.912
  ```
- Финальная модель — **SVM (LinearSVC)** с предобработкой TF-IDF, результат ЛР3
- Эндпоинт доступен в Swagger UI: `http://localhost:8000/docs#/ML%20Inference`

### Пример тестирования через curl

```bash
curl -X POST http://localhost:8000/ml/classify \
  -H "Content-Type: application/json" \
  -d '{"text": "ФРС США сохранила ключевую ставку без изменений"}'
```

Ожидаемый ответ: `category: "Экономика"`

## Шаг 3. Внедрение базового мониторинга

### Реализованные механизмы

#### a) Логирование HTTP-запросов (`logs/app.log`)

В `app/main.py` добавлен `@app.middleware("http")`, который перехватывает каждый запрос:

```
2025-04-10 09:05:13 | INFO  | GET  /health → 200 (3.2ms)
2025-04-10 09:05:14 | INFO  | POST /ml/classify → 200 (48.7ms)
2025-04-10 09:05:15 | WARN  | GET  /sources → 401 (1.1ms)
```

Формат: `[timestamp] | [level] | [METHOD] [path] → [status] ([latency]ms)`

#### b) Эндпоинт `/health` — метрики доступности

```bash
GET http://localhost:8000/health
```

```json
{
  "status": "ok",
  "timestamp": "2025-04-10T09:05:00+00:00"
}
```

Используется как `healthcheck` в `docker-compose.yml`. Бот стартует только после успешного health-check бэкенда.

#### c) Эндпоинт `/metrics` — метрики нагрузки

```bash
GET http://localhost:8000/metrics
```

```json
{
  "uptime_seconds": 3612.4,
  "total_requests": 847,
  "error_count": 2,
  "error_rate_pct": 0.24,
  "requests_per_endpoint": {
    "/health": 120,
    "/ml/classify": 45,
    "/news": 200,
    "/sources": 150,
    "/rules": 90,
    "/auth/login": 37,
    "/metrics": 205
  },
  "collected_at": "2025-04-10T09:01:00+00:00"
}
```

**Собираемые метрики:**
- `uptime_seconds` — время работы сервера с момента запуска
- `total_requests` — всего обработанных запросов
- `error_count` / `error_rate_pct` — число и доля ошибочных ответов (HTTP ≥ 400)
- `requests_per_endpoint` — разбивка hit rate по эндпоинтам

#### d) Логирование инференса (`logs/inference.log`)

Отдельный лог-файл фиксирует каждый вызов `/ml/classify`:

```
2025-04-10 09:05:13 | INFO | text="Биткоин установил..." | category=Криптовалюты | confidence=0.912
2025-04-10 09:05:19 | INFO | text="ФРС сохранила ставку..." | category=Экономика | confidence=0.847
```

## Шаг 4. Контейнеризация и оркестрация

### Созданные файлы

| Файл | Назначение |
|---|---|
| `Dockerfile` | Backend: Python 3.10-slim + FastAPI + ML |
| `bot/Dockerfile` | Telegram-бот |
| `frontend/Dockerfile` | Многоэтапная сборка: Node.js → Nginx |
| `frontend/nginx.conf` | SPA-fallback + проксирование `/api/` на backend |
| `docker-compose.yml` | Оркестрация 3 сервисов |
| `.dockerignore` | Исключение `node_modules`, `venv`, `.env`, `*.db` |
| `.env.example` | Шаблон переменных окружения |

### Сервисы в docker-compose.yml

```
backend   → порт :8000  — FastAPI с ML
frontend  → порт :80    — React/Nginx
bot       →             — Telegram-бот (без внешнего порта)
```

**Особенности конфигурации:**
- `bot` зависит от `backend` с условием `service_healthy` — бот стартует только после прохождения health-check
- Общий том `db_data` монтируется в `/data` у бэкенда и бота (SQLite-файл доступен обоим)
- Том `logs_data` разделяется для сбора всех логов в одном месте
- Frontend-образ использует multi-stage build: сначала `node:20-alpine` собирает статику, затем `nginx:1.25-alpine` её раздаёт

### Команда запуска

```bash
docker-compose up --build
```

Сборка занимает ~2-5 минут при первом запуске (загрузка базовых образов, установка зависимостей Python, npm build).

### Проверка работы

```bash
# Health-check вручную
curl http://localhost:8000/health

# Тестирование инференса
curl -X POST http://localhost:8000/ml/classify \
  -H "Content-Type: application/json" \
  -d '{"text": "Нефть Brent выросла до 90 долларов на фоне решения ОПЕК"}'

# Веб-панель
open http://localhost
```

## Итоги и выводы

В ходе лабораторной работы №4 был завершён полный цикл разработки ML-сервиса:

1. **Инференс-модуль:** реализован REST-эндпоинт `POST /ml/classify` на базе финальной модели SVM (ЛР3) с логированием каждого предсказания.

2. **Интеграция:** все компоненты системы (парсер, ML, API, бот, веб-панель) объединены в единую систему с общей базой данных и разделяемыми логами.

3. **Мониторинг:** внедрены промышленные практики — middleware для логирования HTTP, отдельная метрика инференса, `/health` для проверки доступности и `/metrics` для сбора show нагрузки. Система позволяет в любой момент проверить состояние сервиса без подключения к серверу.

4. **Контейнеризация:** система упакована в 3 Docker-образа и запускается одной командой `docker-compose up --build`. Сервисы изолированы, данные персистентны, настраиваются только через `.env`.

5. **Документация:** создан полноформатный `README.md` с архитектурной диаграммой, инструкцией по запуску, примерами запросов к API и описанием ML-модуля.

**Итоговая точность модели (SVM, CV 5-fold):** 57.4% на 401 примере по 7 категориям финансовых новостей.
